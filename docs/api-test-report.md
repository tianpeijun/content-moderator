## 审核结果说明

| 结果 | 含义 | 是否需要人工审核 |
|------|------|-----------------|
| pass | 内容正常，无违规 | 否，自动通过 |
| reject | 内容明确违规，直接拒绝发布 | 否，自动拒绝 |
| flag | 内容存在风险但不确定，标记待审 | 是，需人工确认后决定通过或拒绝 |
| review | 系统无法判定，提交人工审核 | 是，需人工审核 |

> flag 和 review 都需要人工审核。区别在于：flag 表示系统检测到了可疑内容但不够确定，review 表示系统完全无法判定。reject 是最严格的处理，系统有高置信度认为内容违规。

### 关键发现

1. **多语种支持良好**：系统成功处理了中文、英文、日文、韩文、法文、俄文、阿拉伯文共 7 种语言，以及中英混合语种
2. **违规检测有效**：所有违规内容均被正确标记为 reject 或 flag，无漏检（0 误放）
3. **无误判**：所有正常评论均被正确判定为 pass（0 误杀）
4. **图片审核能力**：系统能识别图片中嵌入的文字信息（联系方式、广告语），并结合文本内容综合判定
5. **命中规则返回**：优化提示词后，违规内容的 matched_rules 正确返回了具体的命中规则名称和处理动作
6. **模型稳定**：Claude Sonnet 4 主模型全程无降级，响应稳定
7. **响应时间可接受**：纯文本平均 2.9 秒，图文混合平均 5.8 秒，满足日处理 2000 条评论的业务需求

### 待优化项

1. 部分违规内容被判定为 flag 而非 reject，可通过调整审核规则的提示词来优化判定阈值
2. 图文混合审核耗时较长（5-12秒），可考虑异步处理或优化图片传输
3. 建议后续使用真实业务数据进行批量测试，进一步验证准确率和召回率
4. 建议为不同业务类型创建专属审核规则，提升审核精准度

## 附录 A：批量测试 Excel 模板说明

### 文件格式要求

- 文件格式：`.xlsx`（Excel 2007+）
- 上传接口：`POST /api/admin/test-suites/upload`（管理后台"批量测试"页面）
- 上传后文件存储位置：S3 TestSuitesBucket（路径：`test-suites/{uuid}/{filename}.xlsx`）

### 必填列定义

| 列名 | 类型 | 必填 | 说明 |
|------|------|------|------|
| 序号 | 数字 | 是 | 测试用例编号，从 1 开始递增 |
| 内容文本 | 文本 | 否* | 待审核的评论文本内容 |
| 图片URL | 文本 | 否* | 待审核的图片 URL（支持 http(s):// 和 s3:// 格式） |
| 期望结果 | 文本 | 是 | 期望的审核结果，仅允许：`pass`、`reject`、`review`、`flag` |
| 业务类型 | 文本 | 否 | 业务场景分类，如"商品评论"、"商家回复"等 |
| 备注 | 文本 | 否 | 测试用例的补充说明 |

> *"内容文本"和"图片URL"至少填写一项。

### 模板示例

| 2 | Great product, highly recommended! | | pass | 商品评论 | 英文正常评论 |
| 3 | この商品はとても良いです。 | | pass | 商品评论 | 日文正常评论 |

### 格式验证规则

系统上传时会自动验证：
1. 文件必须是 `.xlsx` 格式
2. 第一行必须包含所有 6 个必填列名（序号、内容文本、图片URL、期望结果、业务类型、备注）
3. "期望结果"列的值必须是 `pass`、`reject`、`review`、`flag` 之一（不区分大小写）
4. 验证失败时返回具体的错误信息（如"缺少必要列: 图片URL, 备注"或"第3行期望结果值无效: 'block'"）

### 测试报告输出

批量测试完成后，系统生成的报告包含：
- 准确率（Accuracy）、召回率（Recall）、F1 分数
- 混淆矩阵（TP / FP / TN / FN）
- 错误案例列表（期望结果 vs 实际结果）
- 各规则命中分布统计
